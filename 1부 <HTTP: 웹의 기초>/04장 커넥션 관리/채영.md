# 04 커넥션 관리

## 4.1 TCP 커넥션

### 4.1.1 신뢰할 수 있는 데이터 전송 통로인 TCP

TCP는 HTTP에게 신뢰할 만한 통신 방식을 제공. TCP 커넥션의 한쪽에 있는 바이트들은 반대쪽으로 순서에 맞게 정확히 전달됨.

### 4.1.2 TCP 스트림은 세그먼트로 나뉘어 IP 패킷을 통해 전송된다

TCP는 세그먼트라는 단위로 데이터 스트림을 잘게 나누고, 세그먼트를 IP 패킷이라고 불리는 작은 조각을 통해 데이터를 전송함. HTTP는 `IP`,`TCP`, `HTTP`로 구성된 프로토콜 스택에서 최상위 계층임.
![HTTP와 HTTPS 네트워크 프로토콜 스택](https://github.com/learning-with/book-reading-http-reading-perfect-guide/assets/124546770/dc008124-b0a4-4f07-8c51-9ed5eeaa5de2)
HTTP에 보안 기능을 더한 HTTPS는 TLS 혹은 SSL이라 불리기도하며 HTTP와 TCP 사이에 있는 암호화 계층임.

- 각 TCP 세그먼트는 하나의 IP 주소에서 다른 IP주소에서 다른 IP 주소로 IP패킷에 담겨 전달됨.

#### IP패킷에 포함된 것.

- IP 패킷 헤더 (보통 20바이트)
- TCP 세그먼트 헤더(보통 20바이트)
- TCP 데이터 조각(0 혹은 그 이상의 바이트)

### 4.1.4 TCP 소켓 프로그래밍

#### TCP 커넥션 프로그래밍을 위한 공통 소켓 인터페이스 함수들

소켓 API를 사용하면 TCP 종단(endpoint) 데이터 구조를 생성하고, 원격 서버의 TCP 종단에 그 종단 데이터 구조를 연결하여, 데이터 스트림을 읽고 쓸 수 있음.
| 소켓 API 호출 | 설명 |
| ---------------------------- | ---------------------------------------------------------- |
| s=socket(<parameters>) | 연결이 되지 않은 익명의 새로운 소켓 생성 |
| bind(s, <local IP:port>) | 소켓에 로컬 포트 번호와 인터페이스 할당 |
| connect(s, <remote IP:port>) | 로컬의 소켓과 원격의 호스트 및 포트 사이에 TCP 커넥션 생성 |
| listen(s,...) | 커넥션을 받아들이기 위해 로컬 소켓에 허용함을 표시 |
| s2 = accept(s) | 누군가 로컬 포트에 커넥션을 맺기를 기다림 |
| n = read(s,buffer, n) | 소켓으로부터 버퍼에 n바이트 읽기 시도 |
| n = write(s, buffer, n) | 소켓으로부터 버퍼에 n 바이트 쓰기 시고 |
| close(s) | TCP 커넥션을 완전히 끊음 |
| shutdown(s,<side>) | TCP 커넥션의 입출력만 닫음 |
| getsockopt(s, ...) | 내부 소켓 설정 옵션값을 읽음 |
| setsockopt(s, ...) | 내부 소켓 설정 옵션값을 변경 |

TCP API는 기본적인 네트워크 프로토콜의 핸드셰이킹, TCP 데이터 스트림과 IP패킷 간의 분할 및 재조립에 대한 모든 세부사항을 외부로부터 숨김.

## 4.2 TCP의 성능에 대한 고려

HTTP 트랜잭션의 성능은 그 아래 계층인 TCP 성능에 영향을 받음. 기본적인 TCP 성능의 특성을 이해해서 HTTP의 커넥션 최적화 요소들을 더 잘 알게 되고 더 좋은 성능의 HTTP 애플리케이션을 설계하고 구현할 수 있음.

### 4.2.1 HTTP 트랜잭션 지연

요청 과정 중 어떤 지연이 발생할까? 너무 복잡하거나 많은 데이터나 동적인 자원들을 실행하지 않는 이상 대부분 HTTP 지연은 TCP 네트워크 지연 때문에 발생함.

#### HTTP 트랙잭션을 지연시키는 원인

1. URI에 기술되어 있는 호스트를 방문한 적 없을 경우 DNS 이름 분석 인프라를 사용해 URI에 있는 호스트 명을 IP 주소로 변환하는데 수십초의 시간이 걸림
2. 수백 개의 HTTP 트랜잭션이 만들어지면 소요시간이 증가함.
3. 요청 메시지가 인터넷을 통해 전달되고 서버에 의해 처리되는데 까지는 시간이 소요됨.
4. 웹 서버가 HTTP 응답을 보내는 것 역시 시간이 소요됨.

### 4.2.2 성능 관련 중요 요소

- TCP 커넥션의 핸드셰이크 설정
- 인터넷의 혼잡을 제어하기 위한 TCP 느린 시작
- 데이터를 한데 모아 한 번에 전송하기 위한 네이글(nagle)알고리즘
- TCP 편승(piggyback) 확인 응답(acknowledgment)을 위한 확인 응답 지연 알고리즘
- TIME_WAIT 지연과 포트 고갈

### 4.2.3 TCP 커넥션 핸드셰이크 지연

HTTP 트랜잭션이 아주 큰 데이터를 주고받지 않는 평범한 경우, SYN/SYN+ACK 핸드 셰이크가 눈에 띄는 지연을 발생시킴.
TCP 구성으로 인한 지연을 제거하기 위해 HTTP가 이미 존재하는 커넥션을 어떻게 재활용하는지 알아보기.

### 4.2.4 확인 응답 지연

인터넷 자체가 패킷 전송을 보장하지 않음. TCP는 성공적인 데이터 전송을 보장하기 위해 자체적인 확인 체계를 가짐 = TCP 세그먼트 순번과 데이터 무결설 체크섬을 가짐.

### 4.2.5 TCP 느린 시작

TCP 커넥션은 시간이 지나면서 자체적으로 튜닝이 되어 처음에는 커넥션의 최대 속도를 제한하고 데이터가 성공적으로 전송됨에 따라 속도 제한을 높혀가는 것을 말함. 급작스러운 부하와 홈잡을 방지하는데 씀.

### 4.2.6 네이글(Nagle) 알고리즘과 TCP_NODELAY

- 네이글 알고리즘 - 네트워크 효율을 위해 패킷을 전송하기 전에 많은 양의 TCP 데이터를 한 개의 덩어리로 합치는 알고리즘.
- HTTP 애플리케이션은 성능 향상을 위해 HTTP 스택에 TCP_NODELAY 파라미터 값을 설정하여 네이글 알고리즘을 비활성화하기도 함.

### 4.2.7 TIME_WAIT의 누적과 포트 고갈

현대 빠른 라우터들 덕분에 커넥션이 닫힌 후에 중복되는 패킷이 생기는 경우는 거의 없어짐. 하지만 이전 커넥션의 패킷이 커넥션과 같은 연결값으로 생성된 커넥션에 삽입되면 패킷은 중복되고 TCP 데이터는 충돌할 것임. 이때 2MSL의 커넥션 종료 지연이 문제가 되진 않지만 성능 시험 시 문제가 발생할 수 있음.
포트 고갈 문제를 겪지 않더라도 커넥션이 너무 많이 맺거나 대기 상태로 있는 제어 블록이 너무 많아지는 경우 주의할 것!

## 4.3 HTTP 커넥션 관리

### 4.3.1 흔히 잘못 이해하는 Connection 헤더

Connection 헤더의 세가지 종류의 토큰이 전달됨. `HTTP 헤더 필드명`, `임시적인 토큰값`, `close값`

- 커넥션 토큰이 HTTP 헤더 필드 명을 가지고 있으면 해당 필드들의 현재 커넥션만을 위한 정보이므로 다음 커넥션에 전달하면 안됨.

### 4.3.2 순차적인 트랜잭션 처리에 의한 지연

커넥션 관리가 제대로 이루어지지 않으면 TCP 성능은 매우 안좋아짐.
순차적으로 로드하는 방식의 단점은 특정 브라우저에서 객체를 화면에 배치할 경우 객체의 크기를 알아야하기 때문에 모든 객체를 내려받기 전까지 텅 빈 화면을 보여준다는 점.

#### HTTP 커넥션의 성능을 향상시킬 수 있는 방법

- 병렬 커넥션
- 지속 커넥션
- 파이프라인 커넥션
- 다중 커넥션

## 4.4 병렬 커넥션

객체를 하나씩 내려받는 식으로 웹페이지를 보여줄 수 있지만 너무 느림. HTTP는 클라이언트가 여러 개의 커넥션을 맺음으로써 여러 개의 HTTP 트랜잭션을 병렬로 처리할 수 있게함. 병렬커넥션은 여러개의 서버와 맺어질 수 있어야함.

### 4.4.1 병렬 커넥션은 페이지를 더 빠르게 내려받는다

단일 커넥션의 대역폭 제한과 커넥션이 동작하고 있는 시간을 활용하면, 객체가 여러 개 있는 웹페이지를 더 빠르게 내려받을 수 있음.
HTML 페이지를 먼저 내려받고 남은 세 개의 트랜잭션이 각각 별도의 커넥션에서 동시에 처리됨. 이미지들을 병렬로 내려받아 커넥션 지연이 켭쳐짐으로써 총 지연시간이 줄어듬.

### 4.4.2 병렬 커넥션이 항상 더 빠르지는 않다

클라이언트의 네트워크 대역폭이 좁을때는 대부분 시간을 데이터를 전송하는데만 쓸 것임. 제한된 대역폭 내에서 각 객체를 전송받는 것은 느림으로 성능상의 장점이 없어지고 다수의 커넥션이 메모리를 많이 소모하여 자체적인 성능 문제가 발생함. 서버는 특정 클라이언트로부터 과도한 수의 커넥션이 맺어졌을 경우 임의로 끊어버릴 수 있음.

### 4.4.3 병렬 커넥션은 더 빠르게 느껴질 수 있다

병렬 커넥션이 실제호 페이지를 더 빠르게 내려받는 것이 아닐때, 화면에 여러 개의 객체가 동시에 보이면서 내려받고 있는 상황을 보게 될 경우 사용자는 더 빠르게 내려받고 있는 것처럼 느낌.

## 4.5 지속 커넥션

서버에 HTTP요청을 하기 시작한 애플리케이션이 웹페이지 안에 첨부된 이미지 등을 가져오기 위해 서버에 또 요청하는 것을 지역성(site locality)이라함. 처리가 완료된 후에도 계속 연결된 상태로 있는 TCP 커넥션을 지속 커넥션이라고함.

### 4.5.1 지속 커넥션 vs 병렬 커넥션

> #### 병렬 커넥션 단점
>
> - 각 트랜잭션마다 새로운 커넥션을 맺고 끊기 때문에 시간과 대역폭이 소요됨.
> - 각각의 새로운 커넥션은 TCP 느린 시작 때문에 성능이 떨어짐.
> - 실제로 연결할 수 있는 병렬 커넥션의 수에는 제한이 있음.

지속 커넥션은 커넥션을 맺기 위한 사전 작업과 지연을 줄여주고, 튜닝된 커넥션을 유지하며, 커넥션의 수를 줄여줌. 지속 커넥션을 관리 못할 경우 계속 연결된 상태로 수 많은 커넥션이 쌓이게 되어 로컬의 리소스, 원격의 클라이언트와 서버의 리소스에 불필요한 리소스가 발생함. 지속커넥셔은 병렬커넥션과 함께 사용될때 효과적임!

### 4.5.2 HTTP/1.0+의 Keep-Alive 커넥션

초기 지속 커넥션은 상호 운용과 관련된 설계에 문제 있지만, 아직 많은 클라이언트와 서버가 초기 keep-alive 커넥션을 사용중임.

### 4.5.3 Keep-Alive 동작

아직 keep-Alive가 널리 사용중이므로 HTTP 애플리케이션은 그것을 처리할 수 있게 개발해야함.

1. HTTP/1.0 keep-alive 커넥션을 구현한 클라이언트는 커넥션을 유지하기 위해서 요청에 Connection:Keep-Alive헤더를 포함시킴.
2. 요청을 받은 서버는 다음 요청도 커넥션을 통해 받고자 한다면, 응답 메시지에 같은 헤더를 포함시켜 응답함.
3. 응답에 Connection: Keep-Alive 헤더가 없으면, 클라이언트는 서버가 keep-alive를 지원하지 않으며, 응답 메시지가 전송되고 나면 서버 커넥션을 끊을 것이라 추정함.

### 4.5.4 Keep-Alive 옵션

keep-alive 헤더는 커넥션을 유지하기를 바라는 요청일 뿐. 클라이언트나 서버가 keep-alive 요청을 받았다고 해서 무조건 그것을 따를 필요는 없음. keep-alive 헤더 사용은 선택 사항이지만 Connection : keep-alive 헤더가 있을 경우에만 사용 가능.

- keep-alive 동작은 keep-alive 헤더의 쉼표로 구분된 옵션들로 제어할 수 있음.
- `timeout 파라미터`는 커넥션이 얼마나 유지될 것인지 의미함.
- `max 파라미터`는 커넥션이 몇 개의 HTTP 트랜잭션을 처리할 때까지 유지될 것인지를 의미.
- keep-alive 헤더는 진단이나 디버깅을 주 목적으로 하는 처리되지 않는 임의의 속성들을 지원하기도 함. `이름[=값]` 의 형태.

> #### 예시
>
> 서버가 약 5개 추가 트랜잭션이 처리될 동안 커넥션을 유지하거나 2분 동안 커넥션을 유지하라는 내용

```
Connection : Keep-Alive
Keep-Alive : max=5, timeout=120
```

### 4.5.5 Keep-Alive 커넥션 제한과 규칙

- keep-alive는 HTTP/1.0에서 기본으로 사용되지 않음. keep-alive 커넥션 사용시 모든 메시지에 Connection: Keep-Alive 요청 헤더를 보내야함.
- 커넥션을 계속 유지하려면 모든 메시지에 Connection: Keep-alive요청 헤더를 보내야함. 클라이언트가 Connection:Keep-Alive 헤더를 보내지 않으면 서버는 요청 처리 후 커넥션을 끊음. 응답헤더가 없는 것을 보고 커넥션이 끊긴 것을 알 수 있음.
- 커넥션이 끊어지기 전 엔터티 본문에 정확한 Content-Length 값과 함께 멀티 파트 미디어 형식을 가지거나 청크 전송 인코딩으로 인코드 되어야함. 잘못된 Content-Length 값 전송 시 트랜잭션이 끝나는 시점에 기존 메시지의 끝과 새로운 메시지의 시작점을 정확히 알 수 없음.
- 프락시와 게이트웨이는 Connection 헤더의 규칙을 철저히 지켜야함. 프락시와 게이트웨이는 메시지를 전달하거나 캐시에 넣기 전에 Connection 헤더에 명시된 모든 헤더 필드와 Connection 헤더를 제거해야함.
- 멍청한 프락시로 인해 발생할 문제를 예방하기 위해 Connection헤더를 인식하지 못하는 프락시 서버와 연결 금지.

### 4.5.6 Keep-Alive와 멍청한 (dumb) 프락시

#### Connection 헤더의 무조건 전달

프락시는 Connection 헤더를 이해하지 못해 해당 헤더들을 삭제하지 않고 요청 그대로를 다음 프락시에 전달함. Connection 헤더를 이해하지 못하기 때문에 커넥션이 끊어지기를 기다리다 잘못된 통신이 일어나고 브라우저는 자신이나 서버가 타임아웃이 나서 커넥션이 끊어질 때까지 기다림.

#### 프락시와 홉별 헤더

프락시는 Connection헤더와 Connection 헤더에 명시된 헤더들은 절대 전달하면 안됨. ㄷ흔히 잘못 이해하는 Connection 헤더 참고.

### 4.5.7 Proxy-Connection 살펴보기

모든 헤더를 무조건 전달하는 문제를 해결할 수 있는 차선책 = proxy-Connection 헤더 사용하기.

- 프락시가 proxy-Connection 을 전달하더라도 웹 서버는 무시하기 때문에 별 문제가 되지않음.
- 영리한 프락시라면 의미없는 proxy-Connection 헤더를 Connection헤더로 바꿈으로써 원하던 효과를 얻게됨.

### 4.5.8 HTTP/1.1의 지속 커넥션

클라이언트가 응답에 Connection: close 헤더가 없으면 응답 후에도 HTTP/1.1 커넥션을 계속 유지하자는 것으로 추정.

### 4.5.9 지속 커넥션의 제한과 규칙

- Connection: close가 헤더에 포함돼 있다면 추가 요청을 보낼 수 없음.
- 커넥션에 있는 모든 메시지가 자신의 길이 정보를 정확히 가지고 있을 때만 커넥션을 지속시킬 수 있음.
- HTTP/1.1 애플리케이션은 중간에 끊어지는 커넥션을 복구할 수 있어야함.
- 하나의 사용자 클라이언트는 서버의 과부하 방지를 위해 넉넉잡아 두개의 지속 커넥션만 유지해야함.

## 4.6 파이프라인 커넥션

keep-alive의 성능을 더 높여줌. 여러 개의 요청은 응답이 도착하기 전까지 큐에 쌓임. 첫번째 요청이 네트워크를 통해 서버로 전달되면 이어 두번째와 세번째 요청이 전달됨. 이는 대기 시간이 긴 네트워크 상황에서 네트워크상의 왕복으로 인한 시간을 줄여서 성능을 높여줌.

- 지속 커넥션인지 확인하기 전까지 파이프라인을 이으면 안됨.
- 요청/응답 순서 중요.
- 커넥션이 끊어지더라도 완료되지 않은 요청이 파이프라인에 있으면 언제든 다시 요청을 보낼 준비가 돼있어야함.
- POST 요청같이 반복해서 보낼 경우 문제가 생기는 요청은 파이프라인을 통해 보내면 안됨.

## 4.7 커넥션 끊기에 대한 미스터리

### 4.7.1 "마음대로"커넥션 끊기

언제든지 TCP 전송 커넥션을 끊을 수 있음. 하지만 서버가 유휴상태에 있는 커넥션을 끊는 시점에 서버는 클라이언트가 데이터를 전송하지 않을 것이라고 확신을 못함. 클라이언트가 요청 메시지를 보내는 도중 끊어질수도 있음.

### 4.7.2 Connect-Length와 Truncation

HTTP응답은 본문의 정확한 크기 값을 가지는 Content-Length 헤더를 갖고 있어야함.
값이 일치하지 않거나 존재하지 않을 경우, 수신자는 데이터의 정확한 길이를 서버에게 물어봐야함.
만약 수신자가 캐시 프락시일 경우 응답을 캐시하면 안됨. 프락시는 Content-Length를 정정하려하지말고 그대로 전달해야함.

### 4.7.3 커넥션 끊기의 허용, 재시도, 멱등성

응답이 오기 전에 커넥션이 끊기는 경우를 생각하여 멱등성이 지켜져야하는 데이터는 GET, HEAD, PUT, DELETE, TRACE, OPTIONS 메서드를 사용하자.

### 4.7.4 우아한 커넥션 끊기

- 전체 끊기와 절반 끊기
- TCP 끊기와 리셋 에러
- 우아하게 커넥션 끊기 : 애플리케니샨 자신의 출력 채널을 먼저 끊고 다른 쪽에 있는 기기 출력 채널이 끊기는 것을 기다리는 것.
